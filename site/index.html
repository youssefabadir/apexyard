01 / LAYERS

Five layers, one stack

Five layers stack together: roles, workflows, hooks, skills, and the portfolio registry itself. Even with one repo, every layer works the same.

01 — roles

Roles

19 role definitions across 5 departments. Activation-triggered with CAN / CANNOT boundaries. "Act as the QA Engineer" picks up the full mental model.

02 — workflows + templates

Workflows + templates

7-phase SDLC, code review, deployment. Seven templates — PRD, tech design, ADR, AgDR, migration AgDR, C4 L1 + L2.

03 — hooks + rules

Hooks + rules

18 shell hooks enforce the SDLC mechanically: ticket-first, migration gate, two-marker merge gates (Rex + CEO), design review, red-CI block, upstream drift banner. Eight modular rule files.

04 — agents + skills

Agents + skills

Five sub-agents (Rex, Shield, Guardian, PR Manager, Ticket Manager). 32 slash commands covering SDLC, portfolio rollups, adoption, and an 8-dimension /launch-check.

05 — portfolio

A portfolio, not a single repo

Your ops repo is a fork of apexstack. apexstack.projects.yaml at the root is the registry. Every portfolio skill — /projects, /inbox, /status, /tasks, /stakeholder-update — aggregates across it. One fork, N products, one inbox.

02 / WALKTHROUGHS

What it actually feels like to use

Three flows — one for a feature, one for the portfolio, one for a risky change. Each starts with a specific pain and ends with the exact gates that handled it.

  ideas-backlog.md · PRD-0012 · AgDR-0021

› /start-ticket 178     (labels the session)
# code, tests, push…

› gh pr create
  auto-code-review.sh: "Rex REQUIRED before merge"

  APPROVED at c8b736f · marker written

› /approve-merge 178     (per-PR CEO nod)

One trail: 6 skills · 2 agents · 4 hooks · 1 AgDR · 1 PRD. Two independent approvals (Rex + CEO), both SHA-bound. The marker-based merge gate is a shell hook, not a social convention.

03 / PIPELINES

Drop-in CI workflows

Seven GitHub Actions workflows live at golden-paths/pipelines/. Copy whichever you want into your project's .github/workflows/ — no boilerplate to write, no configuration to wrestle with.

ci.yml
Combined pipeline — code quality + security + dependencies in one file. Start here.

Semgrep SAST (OWASP Top 10), npm audit, TruffleHog secrets detection, optional CodeQL. Fails on critical or high severity.

Weekly vulnerability scan (Mondays 09:00 UTC) plus on every package.json change. Auto-creates a security issue when something critical lands.
Guardian

Enforces a ticket ID in every PR title. Default: GitHub Issues (#58). Also accepts any uppercase tracker prefix.
governance
review-check.yml
Blocks merge if Rex hasn't reviewed the latest commit on the PR. Compares review SHAs against HEAD.
Rex (verification)
seo-check.yml
SEO scoring for content files — H1, meta description, length, headings, alt text, internal links. For docs and blog sites.

04 / INSTALL

Fork, track upstream, start working

Three steps. The fork is your ops repo — no nested installs, no symlinks. Config (onboarding.yaml) and the registry (apexstack.projects.yaml) are handled by /setup and /handover from inside Claude Code — you never open them by hand.

01 — fork + clone

Fork apexstack on GitHub

Star + fork on GitHub, or one-command via the GitHub CLI. Rename the fork to your-org/ops if you prefer — GitHub handles the rename cleanly.

gh repo fork me2resh/apexstack --clone
cd apexstack
02 — track upstream

Add the upstream remote

Once. The session-start drift banner tells you when you're behind; /update does the sync-branch-and-PR dance for you.

git remote add upstream \
  https://github.com/me2resh/apexstack.git
03 — start working

Open Claude Code in the fork

Everything loads automatically — CLAUDE.md is the entry point. Run /setup once to configure your company + stack, /handover <repo> to bring each project under management, then reach for the portfolio skills.

/setup             # one-time — company + tech stack → onboarding.yaml
/handover <repo>   # adopt a project → registry + stub L2 C4 diagram
/inbox             # PRs, issues, comments needing your eyes
/status            # git + CI snapshot per project
/launch-check      # 8-dimension production-readiness audit
05 / SCOPE

Opinionated by design

Every reusable framework has to draw a line between "this is what we do" and "this is yours to figure out". ApexStack draws the line where it makes the most difference for engineering teams.

+ what apexstack does

  • Enforces ticket-first development via GitHub Issues per project — with a stricter migration-gate variant on schema / migration-path edits
  • Requires explicit per-PR CEO approval before any merge — marker is SHA-bound, invalidates on every new commit
  • Blocks dangerous git operations (add -A, push to main, secrets in commits, red CI, unreviewed merges via either gh pr merge or raw gh api)
  • Forces structured decisions via /decide → AgDR audit trail (plus a dedicated migration AgDR with rollback / downtime / consumers / observability)
  • Ships an opinionated 6-gate workflow from PRD to QA + a migration sub-workflow
  • Aggregates a whole portfolio into one inbox — /inbox, /status, /tasks read a single registry file
  • Tells you when the fork is behind upstream (session-start drift banner), then syncs it in one skill invocation
  • Mandates a glossary in every PR — knowledge stays in the team
  • Defaults to TypeScript, DDD, vitest, GitHub Actions — but stays editable per project via .claude/project-config.json

× what apexstack does NOT do

  • Replace your team — humans still write code, ship product, talk to customers
  • Lock you into a SaaS — every file is markdown or shell, fork freely, diff any decision in git
  • Hide its opinions — read the rules, disagree, edit them (and file a PR upstream if you want them changed for everyone)
  • Auto-run merges when you say "go" on a plan — per-PR approval is discrete by design
  • Impose a central tracker — issues live in each project's own GitHub repo, not in apexstack
  • Pretend to be language-agnostic — TypeScript-leaning by default; path patterns and templates are swappable
  • Ship a UI — this is a stack of files, not an app
  • Compete with Claude Code's built-ins — it extends them via hooks, skills, and sub-agent definitions

Stop reinventing
the same processes.

ApexStack is one fork, one clone, and one config file away. It's MIT licensed, plain markdown, and edits are encouraged. If you change your mind, you delete the directory and you're back where you started.
